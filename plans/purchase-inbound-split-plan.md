# 进货单与入库单拆分方案

## 背景

当前系统已经切换为 SQLite 本地数据库，商品条形码已经接入 `products.barcode`，库存单件也已经通过
`inventory_items.serial_number` 支持序列号记录。

现有“物品入库”功能实际把两个业务动作合并在一起：

- 记录向供应商进货的商品、数量、价格，并通过付款记录追踪付款金额、账号和时间
- 提交后直接生成库存单件并增加库存

这会导致“下了进货订单但货还没到”“分批到货”“少发/多发”“从非进货单来源补库存”等场景难以表达。因此后续需要把
“进货单”和“入库单”拆成两个独立概念。

## 当前现状

### 已有能力

- `products` 已有 `barcode` 字段，并通过唯一索引保证非空条形码不重复。
- `inventory_items` 已有 `serial_number` 字段，并通过唯一约束保证序列号不重复。
- 入库页已经可以一次输入多个序列号。
- 入库提交会为每个数量生成一条 `inventory_items`。
- 库存数量通过统计 `inventory_items.status = 'in_stock'` 回写到 `products.stock_quantity`。
- 账款管理当前从 `inbound_orders.is_paid = 0` 统计应付款。

### 主要问题

- 当前没有真正的进货单。
- 当前入库单同时承担采购订单、实际入库和应付款凭证三种含义。
- 入库时序列号数量和入库数量没有联动，也缺少强校验。
- 账款应付款挂在入库单上，后续分批入库、部分付款和先付款后到货会比较别扭。

## 目标业务模型

```text
产品型号 products
  管商品名称、条形码、分类、参考价等型号级信息

进货单 purchase_orders
  管向供应商订了什么，不增加库存

入库单 inbound_orders
  管实际到了什么，提交后生成库存

库存单件 inventory_items
  管每一件真实库存，序列号落在这里
```

## 进货单

进货单代表“向供应商下了采购订单”。

### 业务规则

- 进货单不增加库存。
- 进货单明细依赖商品名称、条形码、商品型号和采购数量。
- 进货时通常还没有实际库存件，因此不录入序列号。
- 一张进货单可以分多次生成入库单。
- 进货单需要维护已入库数量，用于判断是否已完成。
- 进货单单号第一版使用自增规则，不允许手动修改。
- 运费和杂费由用户手动输入，计入进货单应付款和采购成本汇总。

### 状态建议

```text
draft           草稿
ordered         已下单
partial_inbound 部分入库
completed       已完成
cancelled       已取消
```

### 主表字段建议

```text
id
supplier_id
status
ordered_at
expected_inbound_at
shipping_fee_cents
misc_fee_cents
note
created_at
updated_at
```

说明：第一版可以直接使用 `id` 作为进货单号，并在页面展示为 `#id` 或 `JH-{id}`。如果后续需要更复杂的单号格式，再增加
`order_no`。

### 付款记录表

第一版不再使用 `is_paid` 开关，也不把累计已付款金额直接存在 `purchase_orders` 上。付款使用独立流水表：

```text
purchase_payments
  id
  purchase_order_id
  amount_cents
  payment_account
  paid_at
  status
  voided_at
  void_reason
  note
  created_at
  updated_at
```

字段含义：

- `amount_cents`：本次付款金额，必须大于 0。
- `payment_account`：本次付款账号或付款渠道，选填。
- `paid_at`：本次付款时间，默认当前时间，但允许手动调整。
- `status`：付款记录状态，`active` 表示有效，`voided` 表示已作废。
- `voided_at`：作废时间。
- `void_reason`：作废原因。
- `note`：本次付款备注，选填。

付款状态由付款流水汇总得到，不需要手动存储：

```text
商品小计 = SUM(ordered_quantity * purchase_price_cents)
采购退货金额 = SUM(已完成采购退货明细金额)
应付款 = 商品小计 + shipping_fee_cents + misc_fee_cents - 采购退货金额
已付款 = SUM(status = 'active' 的 purchase_payments.amount_cents)
已退款 = SUM(status = 'active' 的 purchase_refunds.amount_cents)
净付款 = 已付款 - 已退款
待付款 = MAX(应付款 - 净付款, 0)
待退款 = MAX(净付款 - 应付款, 0)

净付款 = 0                 -> 未付款
0 < 净付款 < 应付款         -> 部分付款
净付款 = 应付款             -> 已结清
净付款 > 应付款             -> 待退款
```

第一版不考虑主动超付和预付款，默认不允许新增付款后净付款大于当前应付款。因采购退货导致净付款大于调整后应付款时，不视为录入错误，而是展示为待退款。

说明：系统不是实时对接银行或支付平台的收付款系统，付款通常由人工在下班后统一上账，或者下单后手动录入。因此第一版只记录已确认发生的付款，不处理支付中、预付款、超付和供应商余额等复杂状态。

### 明细字段建议

```text
id
purchase_order_id
product_id
ordered_quantity
received_quantity
purchase_price_cents
note
created_at
updated_at
```

### 编辑规则

进货单允许编辑，但需要保护库存和财务一致性。

未入库的进货单：

- 可以新增物品。
- 可以编辑商品、数量、采购单价、运费、杂费、预计到货时间和备注。
- 可以删除未入库明细。

部分入库的进货单：

- 可以新增物品。
- 已入库明细不允许修改商品和采购单价。
- 已入库明细的采购数量不能改到小于已入库数量。
- 如果同一商品后续采购价不同，建议新增一条明细，不直接修改已入库明细价格。
- 运费、杂费、预计到货时间和备注可以编辑。

已完成的进货单：

- 不允许修改商品明细、数量和采购单价。
- 可以编辑运费、杂费和备注。

付款相关信息不通过进货单编辑表单直接修改。付款只能通过新增付款记录或作废付款记录处理。

任何编辑如果导致调整后应付款小于有效净付款，系统应阻止保存，并提示先作废或调整相关付款/退款记录。

## 入库单

入库单代表“实际收到货并确认入库”。

### 业务规则

- 入库单提交后生成 `inventory_items`。
- 第一版入库单来源只支持“从进货单导入”和“历史库存”。
- 来源于进货单时，本次入库数量不能超过进货单剩余未入库数量。
- 来源于进货单时，导入前可以选择本次导入数量，默认使用剩余可入库数量。
- 来源于进货单时，选择进货单后自动回填商品明细、采购单价和剩余可入库数量。
- 来源于进货单时，生成入库明细后，商品、采购单价和入库数量不允许手动改；只允许调整每条商品项的序列号管理开关、序列号、质保和备注等入库参数。
- 入库单提交后不允许修改商品明细、数量、成本价和序列号。
- 来源于进货单的入库单不再维护付款信息，付款统一在进货单上维护。
- 历史库存入库第一版不维护付款信息；如果需要产生应付款，必须先走进货单。

### 来源类型建议

```text
purchase_order  来自进货单
opening_stock   期初库存
```

说明：`opening_stock` 第一版用于历史库存、期初库存和无应付款库存补录。手动入库和调账入库暂不单独开放来源，后续如有需要再扩展。

### 主表字段调整建议

在现有 `inbound_orders` 基础上增加：

```text
source_type
purchase_order_id
status
```

说明：第一版可以直接使用 `id` 作为入库单号，并在页面展示为 `#id` 或 `RK-{id}`。如果后续需要更复杂的单号格式，再增加
`inbound_no`。

### 明细字段调整建议

在现有 `inbound_order_items` 基础上增加：

```text
purchase_order_item_id
serial_tracking_enabled
```

其中 `serial_tracking_enabled` 是本次入库明细的快照字段，用于记录这批库存入库时是否按序列号管理。

## 采购退货和换货

采购退货和换货必须先联动入库单或库存单件，不能只在进货单上做财务标记。

### 业务规则

- 已入库商品需要退货或换货时，先在入库单或库存单件侧选择要处理的库存。
- 只有 `inventory_items.status = 'in_stock'` 的库存允许采购退货。
- 已售出库存不能直接采购退货，需要先处理销售退货或其他库存回流流程。
- 退货或换货必须填写原因。
- 采购退货成功后，对应库存件标记为 `returned`，并重算商品库存数量。
- 进货单页面展示关联退货记录和退货金额，但不直接跳过库存侧做退货。

### 数据表建议

```text
purchase_returns
  id
  purchase_order_id
  inbound_order_id
  type
  reason
  status
  created_at
  updated_at
```

说明：第一版可以直接使用 `id` 作为退货单号，并在页面展示为 `#id` 或 `TH-{id}`。如果后续需要更复杂的单号格式，再增加
`return_no`。

```text
purchase_return_items
  id
  purchase_return_id
  purchase_order_item_id
  inbound_order_item_id
  inventory_item_id
  product_id
  purchase_price_cents
  created_at
```

字段说明：

- `type`：`return` 表示退货，`exchange` 表示换货。
- `reason`：退货或换货原因，必填。
- `status`：第一版可先使用 `completed`，后续如需审核流程再扩展。

### 换货处理

第一版建议把换货拆成两个动作：

```text
旧库存采购退货出库
新库存重新入库
```

如果换货没有差价，新入库可以来源于原进货单或退货记录，采购单价沿用原成本。

如果换货存在差价，建议通过新增进货明细或新的进货单处理差额，避免把换货做成隐形改价。

### 供应商退款记录

如果采购退货后已经付款金额大于调整后应付款，进货单和账款页面展示待退款。

收到供应商退款后，登记退款记录：

```text
purchase_refunds
  id
  purchase_order_id
  purchase_return_id
  amount_cents
  refund_account
  refunded_at
  status
  voided_at
  void_reason
  note
  created_at
  updated_at
```

规则：

- `amount_cents`：本次退款金额，必须大于 0。
- `refund_account`：退款到账账号或渠道，选填，第一版自由输入。
- `refunded_at`：退款时间，默认当前时间，可手动调整。
- 退款记录提交后不允许编辑，只允许作废。
- 作废退款记录时必须填写作废原因。
- 新增退款后总有效退款金额不能大于待退款金额。

## 序列号管理

条形码和序列号需要严格区分：

- 条形码是商品型号维度信息，落在 `products`。
- 序列号是库存单件维度信息，落在 `inventory_items`。
- 进货依赖商品名称和条形码。
- 入库依赖实际库存件和序列号。

### 已确认方案：开关放在入库明细并存库

第一版在 `inbound_order_items` 增加：

```text
serial_tracking_enabled INTEGER NOT NULL DEFAULT 0
```

含义是本次入库的这一条明细是否启用序列号管理。

确认理由：

- 序列号是在入库时录入，开关放在入库明细更贴近实际操作。
- 序列号本身选填，但开启序列号管理后必须变为必填，因此开关必须作为历史业务事实存库。
- 同一个商品不同批次可以灵活处理，避免商品档案规则过早限制业务。
- 提交后的入库单即使页面规则变化，也能还原当时是否强制按序列号入库。

商品档案第一版暂不增加“默认启用序列号管理”字段。后续如果发现入库人员容易漏开，再补一个商品默认建议值：

```text
products.serial_tracking_default
```

这个字段只作为入库时的默认值，不作为最终业务事实。最终业务事实仍以
`inbound_order_items.serial_tracking_enabled` 为准。

## 入库序列号交互规则

序列号管理开关以入库商品项为单位设置，并存入 `inbound_order_items.serial_tracking_enabled`。

### 从进货单导入

数量来自进货单剩余可入库数量，不由序列号反推。

开启序列号管理时：

- 序列号必填。
- 序列号个数必须等于该商品项入库数量。
- 如果入库数量为 10，但只填写 2 个序列号，保存时报错。
- 每个序列号生成一条 `inventory_items`。
- 同一张入库单内序列号不能重复。
- 数据库中已存在的序列号不能重复。

关闭序列号管理时：

- 序列号选填。
- 如果填写序列号，已填写序列号数量不能超过该商品项入库数量。
- 未填写序列号的库存件生成 `serial_number = null`。

### 历史库存入库

历史库存没有进货单数量约束。

开启序列号管理时：

- 序列号必填。
- 数量输入禁用。
- 数量自动等于序列号个数。
- 每个序列号生成一条 `inventory_items`。
- 同一张入库单内序列号不能重复。
- 数据库中已存在的序列号不能重复。

关闭序列号管理时：

- 数量可以手动填写。
- 序列号选填。
- 如果填写序列号，已填写序列号数量不能超过数量。
- 未填写序列号的库存件生成 `serial_number = null`。

### 从进货单入库事务

`POST /api/purchase-orders/[id]/receive` 需要使用 SQLite transaction：

```text

校验序列号规则
创建 inbound_orders
创建 inbound_order_items
创建 inventory_items
更新 products.stock_quantity
更新 purchase_order_items.received_quantity
更新 purchase_orders.status
```

## 页面设计

仓库管理菜单建议调整为：

```text
产品型号
物品列表
进货商家
进货单
入库单
```

### 进货单页面

核心功能：

- 新增进货单
- 编辑草稿或未完成进货单
- 取消进货单
- 查看进货单详情
- 从进货单生成入库单
- 查看付款记录
- 登记一笔付款

商品选择：

- 支持商品名称搜索
- 支持条形码搜索
- 下拉展示分类、商品名称和条形码

表单字段：

- 进货商家
- 下单时间
- 预计到货时间
- 运费
- 杂费
- 初始付款金额，默认 0
- 初始付款账号
- 初始付款时间
- 初始付款备注
- 进货明细
- 备注

底部汇总分为两个区域。

商品汇总：

- 明细数
- 采购总数量
- 已入库数量
- 未入库数量
- 商品小计

资金汇总：

- 商品小计
- 采购退货金额
- 运费
- 杂费
- 应付款合计
- 已付款
- 已退款
- 待付款
- 待退款

如果应付款等于净付款，待付款可以不展示金额，改为展示“已结清”。如果因退货导致净付款大于应付款，则展示“待退款”。

付款记录：

- 金额
- 付款账号
- 付款时间
- 状态
- 备注
- 创建时间
- 作废时间
- 作废原因

创建进货单时如果初始付款金额大于 0，则同步创建一条 `purchase_payments`。如果初始付款金额为 0，则不创建付款记录。

### 入库单页面

核心功能：

- 从进货单入库
- 历史库存入库
- 查看入库单详情
- 提交后仅允许编辑备注
- 采购退货

创建入库单时先选择入库来源：

```text
从进货单导入
历史库存
```

从进货单入库时：

- 选择来源后展示进货单选择框。
- 自动带出供应商、商品、采购单价、剩余未入库数量。
- 本次导入数量默认等于剩余未入库数量，但导入前可以调整，且不能超过剩余数量。
- 生成入库明细后，商品、采购单价和数量不允许手动修改。
- 每个商品项都有自己的序列号管理开关。
- 序列号管理开启后，序列号必填，序列号个数必须等于该商品项入库数量。
- 如果导入数量为 10，但只录入 2 个序列号，则保存时报错。
- 序列号管理关闭后，序列号选填；如果填写序列号，已填写数量不能超过该商品项入库数量。

历史库存入库时：

- 手动选择供应商和商品。
- 手动填写成本价、质保、备注等入库信息。
- 每个商品项都有自己的序列号管理开关。
- 序列号管理开启后，数量由序列号个数联动。
- 序列号管理关闭后，数量手动填写，序列号选填。

费用和付款：

- 从进货单导入的入库单不展示运费、杂费和付款字段。
- 历史库存入库不展示运费、杂费和付款字段。
- 费用和应付款统一在进货单处理。

## 财务口径

建议应付款从进货单产生，而不是从入库单产生。

原因：

- 欠供应商的钱通常在采购确认或下单时形成。
- 入库只是货到了，不一定代表此时才产生应付款。
- 分批入库时，如果应付款挂在入库单上，会把同一笔进货拆成多笔账。
- 部分付款需要金额维度，不能再用简单的已付款开关表达。

后续账款管理建议：

- 应付款查询 `purchase_orders` 中 `待付款 > 0` 的进货单。
- 账款列表展示应付款、已付款、待付款和最近付款账号。
- 账款页面支持登记一笔付款，本质上新增 `purchase_payments`。
- 每笔付款记录都必须有金额和付款时间，付款账号选填。
- 付款账号第一版自由输入，后续 TODO 可升级为资金账户字典。
- 付款记录录错后走作废流程，保留原始记录和作废痕迹，不物理删除。
- 付款记录提交后不允许编辑，只允许作废。
- 作废付款记录时必须填写作废原因。
- 采购退货后如果产生待退款，账款页面支持登记一笔供应商退款，本质上新增 `purchase_refunds`。
- 退款记录规则与付款记录一致：提交后不可编辑，只允许作废，作废原因必填。
- 入库单主要影响库存，不直接作为应付款来源。
- 旧入库单可以迁移生成已完成进货单，用于兼容历史数据。

账款页面的付款输入使用“本次付款金额”模型：

```text
应付款 1000，已付款 200，待付款 800
本次付款 300
提交后已付款 = 500，待付款 = 500
```

第一版所有已付款金额均从有效 `purchase_payments` 汇总计算，不在 `purchase_orders` 上额外维护冗余金额字段。

## 历史数据兼容

当前已有的 `inbound_orders` 可以迁移成“已完成进货单 + 已完成入库单”。

迁移规则：

```text
每张旧 inbound_orders -> 生成一张 purchase_orders
每条旧 inbound_order_items -> 生成一条 purchase_order_items
purchase_order_items.ordered_quantity = 原 quantity
purchase_order_items.received_quantity = 原 quantity
purchase_orders.status = completed
如果原 inbound_orders.is_paid = 1，则生成一条 purchase_payments
历史付款 amount_cents = 商品小计 + 运费 + 杂费
历史付款 paid_at = 原 inbound_orders.inbound_at 或 created_at
历史付款 payment_account = NULL
历史付款 status = active
历史付款 note = 历史入库单迁移生成
如果原 inbound_orders.is_paid = 0，则不生成 purchase_payments
inbound_orders.purchase_order_id = 新 purchase_orders.id
inbound_orders.source_type = purchase_order
```

## 验收用例

- 创建进货单后，库存数量不变化。
- 进货单可以按商品名称搜索商品。
- 进货单可以按条形码搜索商品。
- 进货单初始付款金额默认是 0。
- 进货单资金汇总正确展示商品小计、运费、杂费、应付款、已付款和待付款。
- 进货单创建时初始付款金额大于 0，会生成一条付款记录。
- 每条付款记录包含金额、付款账号、付款时间和备注。
- 进货单已付款金额等于应付款时，资金汇总展示已结清。
- 新增付款后净付款不能大于应付款。
- 付款记录提交后不能编辑，只能作废。
- 作废付款记录必须填写作废原因。
- 从进货单生成入库单后，库存数量增加。
- 从进货单导入入库单时，商品、采购单价和数量自动回填且不可手动修改。
- 从进货单导入入库单时，每个商品项可以单独设置序列号管理开关。
- 从进货单导入且开启序列号管理时，序列号个数必须等于该商品项入库数量。
- 历史库存入库且开启序列号管理时，数量自动等于序列号个数。
- 入库单提交后核心字段只读，仅允许编辑备注。
- 进货单支持部分入库，状态变为 `partial_inbound`。
- 全部入库后，进货单状态变为 `completed`。
- 开启序列号管理时，未填序列号不能提交。
- 重复序列号不能提交。
- 从进货单导入且关闭序列号管理时，序列号选填，已填序列号数量不能超过商品项入库数量。
- 历史库存入库且关闭序列号管理时，数量手动填写，序列号选填。
- 应付款来自进货单，而不是入库单。
- 账款管理只展示待付款大于 0 的进货单。
- 在账款管理登记一笔付款后，进货单待付款同步减少，并能查到付款明细。
- 已入库商品采购退货必须从入库单或库存单件侧发起。
- 采购退货必须填写原因。
- 采购退货后库存数量减少，进货单应付款同步扣减退货金额。
- 因采购退货产生待退款时，账款页面展示待退款。
- 登记供应商退款后，待退款同步减少，并能查到退款明细。
- 退款记录提交后不能编辑，只能作废。
- 作废退款记录必须填写作废原因。
- 数据导出导入后，进货单、入库单、库存单件关系不丢失。

## 后续待讨论事项

这些点不影响先搭主流程，但建议在正式开发前逐个确认。

### 序列号管理开关

已确认：

- 第一版放在入库明细并存库。
- 真正业务事实使用 `inbound_order_items.serial_tracking_enabled`。
- 商品档案暂不强制增加默认开关。
- 开启后序列号必填，并校验全库唯一。

后续可选增强：

- 商品档案增加“默认启用序列号管理”的建议值，用于入库时自动带出默认开关。
- 如果入库明细关闭序列号管理但填写了部分序列号，销售出库时可以优先展示有序列号库存。

### 付款金额模型

已确认：

- 第一版采用“本次付款金额”模型。
- 每次付款都写入 `purchase_payments`。
- 每条付款记录保存金额、付款账号、付款时间和备注。
- 进货单已付款金额从付款记录汇总得到。
- 账款页面登记的是一笔新的付款记录，而不是覆盖累计金额。
- 第一版不考虑主动超付和预付款，新增付款后净付款不能大于当前应付款。
- 付款账号第一版自由输入，后续 TODO 升级为资金账户字典。
- 付款记录录错后只能作废，不物理删除，已作废记录不计入已付款汇总。
- 付款记录提交后不允许编辑。
- 作废付款记录时必须填写作废原因。

示例：

```text
应付款 1000，已付款 200，待付款 800
本次付款 300
提交后已付款 = 500，待付款 = 500
付款明细新增一条 300 的记录
```

后续可选增强：

- 资金账户字典。
- 付款记录导出、筛选和按账号统计。

### 历史库存入库和财务

已确认：

- 从进货单入库不再维护付款。
- 历史库存、期初库存和无应付款库存补录主要影响库存。
- 第一版如果需要产生应付款，必须先走进货单，再由进货单生成入库单。
- 不建议让历史库存入库直接带付款字段，避免绕过账款主流程。

### 运费和杂费归属

已确认：

- 进货单上的运费和杂费计入应付款。
- 运费和杂费由用户手动输入，算作采购相关成本。
- 入库单只记录实际库存入库，第一版不把运费和杂费分摊进库存单件成本。
- 分批入库时，运费和杂费仍跟进货单走。

后续可选增强：

- 按数量或金额把运费、杂费分摊进库存成本。
- 将进货单运费、杂费同步拆到经营成本明细中做更细统计。

### 退货和退款

已确认：

- 已入库商品退货或换货必须先联动入库单或库存单件。
- 只有仍在库的库存件可以采购退货。
- 采购退货必须填写原因。
- 采购退货完成后减少库存，并扣减进货单应付款。
- 如果退货后净付款大于应付款，则展示待退款。
- 收到供应商退款后登记退款记录。
- 退款记录提交后不允许编辑，只允许作废，作废原因必填。
- 换货第一版拆成“旧库存采购退货出库 + 新库存重新入库”两个动作。

后续可选增强：

- 换货差价自动结算。
- 供应商余额管理。
- 采购退货审核流程。

### 单号规则

已确认：

- 第一版单号使用自增规则。
- 进货单可以直接展示 `#id` 或 `JH-{id}`。
- 入库单可以直接展示 `#id` 或 `RK-{id}`。
- 第一版不允许手动修改单号。

后续可选增强：

- 日期前缀单号，例如 `JH-20260603-0001`。

### 编辑权限

已确认：

- 未入库进货单允许编辑商品、数量、采购单价、运费、杂费、预计到货时间和备注。
- 部分入库后，已入库明细不允许修改商品和采购单价；采购数量不能小于已入库数量。
- 已完成进货单不允许修改商品明细、数量和采购单价。
- 运费、杂费和备注可以编辑，但不能导致调整后应付款小于有效净付款。
- 付款相关信息不在进货单编辑表单里直接修改，只能通过付款/退款记录新增或作废处理。
