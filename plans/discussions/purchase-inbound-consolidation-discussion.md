# 进货、入库与采购退货讨论稿

## 背景

当前系统已经把“进货单”和“入库单”拆开：

- `purchase_orders` 表达向商家下了什么货、采购数量、采购价格、运费和付款。
- `inbound_orders` 表达实际到货入库的批次，并在提交后生成 `inventory_items`。
- 采购退货基于入库单下仍在库的 `inventory_items` 处理，成功后库存件状态变为 `returned`。
- 采购退货还需要表达退货寄回、退款延迟、退货运费等过程信息，因此需要采购退货记录承接后续维护。

从数据模型看，这个拆分是合理的；但从用户操作看，普通采购流程仍然需要在“进货订单”和“入库单”之间来回跳转，容易让用户觉得入库单是一个独立业务入口。

## 当前判断

入库单不应该作为普通采购的主入口。

原因：

- 进货单列表的状态列建议改名为 `货物状态`，只表达货有没有下单、有没有入库。
- 进货单列表的资金汇总建议改名为 `资金状态`，只表达钱有没有结清。
- 货物状态建议表达 `预下单`、`已下单`、`部分入库`、`已入库`、`已取消`。
- 原来的 `已完成` 在拆分后等同于 `已入库`，不再保留单独状态。
- 下单后并不应该直接生成入库单，否则会把“待到货”误表达成“已经产生入库批次”。
- 入库单一旦创建，就会生成真实库存件；它更像实际入库流水，而不是待办单。
- 采购退货必须落到具体库存件或入库批次上，不能只在进货单上做财务标记。
- 采购退货从入库单发起，但退货过程维护应放在采购退货记录里，不塞进入库单列表操作。

因此更合理的产品形态是：

```text
进货/退货页 = 采购工作台
入库单/入库记录 = 进货单下的实际到货流水与退货发起入口
采购退货记录 = 退货过程维护与退款跟踪
```

## 建议方向

### 1. 以进货/退货页作为主工作台

`app/admin/dashboard/warehouse/purchase-orders/` 建议在菜单上改名为 `进货/退货`，作为采购业务的主要入口。

页面内部使用 Tab 区分：

- `进货`：维护进货单、货物状态、资金状态和付款。
- `退货`：维护采购退货记录、货物状态、资金状态、退货运费和退货物流。

`进货/退货` 的 Tab 形态可以作为长期形态保留，不需要再拆成两个独立菜单。

进货 Tab 覆盖这些动作：

- 新增进货单
- 编辑进货单
- 记录付款
- 到货入库
- 查看入库记录

退货 Tab 覆盖这些动作：

- 查看采购退货记录
- 更新货物状态
- 记录或查看退款
- 维护退货运费、运费承担方和退货物流信息

采购退货的发起入口仍在入库单页面，不在进货 Tab 直接发起。

用户的主路径变成：

```text
新增进货单（预下单） -> 确认下单 -> 到货时点跳转入库单页完成“入库” -> 付款仍在进货单维护

入库单页面 -> 基于具体入库单发起退货 -> 创建采购退货记录 -> 在进货/退货页的退货 Tab 维护货物状态、资金状态和退货运费
```

### 2. 保留 inbound_orders 作为底层入库批次表

不建议删除 `inbound_orders`。

它仍然有清晰职责：

- 记录每次真实到货时间。
- 支持一张进货单分多次入库。
- 记录每批次的入库商品、数量、成本价快照、序列号和质保信息。
- 支持对某一批次下仍在库的库存件做采购退货。
- 支持历史库存或期初库存补录。

也就是说，UI 可以弱化“入库单页面”，但数据模型仍然需要入库单。

### 3. 增加采购退货记录作为过程单据

建议增加采购退货记录，用于承接退货后的过程维护。

退货记录不是从进货单直接发起，而是从入库单页面的具体入库记录发起。

它的职责是：

- 记录退了哪些商品和库存件。
- 记录退货金额。
- 记录退货运费和运费承担方。
- 跟踪退货货物状态，页面列名可展示为 `货物状态`。
- 跟踪退款状态，页面列名可展示为 `资金状态`。
- 关联进货单、入库单和退货资金记录。

退货运费承担方建议：

```text
我方 / 商家 / 平摊
```

如果选择 `平摊`，需要记录我方承担金额和商家承担金额，方便后续统计。

采购退货记录的状态也建议拆成货和钱两个维度：

```text
货物状态：待寄回 / 已寄回 / 商家已收货 / 已取消
资金状态：未退款 / 部分退款 / 已退款
```

采购退货记录可以放在 `进货/退货` 页的 `退货` Tab 中维护。入库单页面只负责从具体入库批次发起退货，以及跳转查看关联退货记录。

### 4. 入库单页面保留为入库记录和入库执行页

入库菜单需要保留，因为入库单页面仍然承接真实入库、入库记录查询，以及从具体入库批次发起采购退货。

菜单名称保留 `入库单`，不改名，也不隐藏入口。

它的定位是：

- 从进货单跳转过来新增入库。
- 查询全局入库记录。
- 查看某张进货单下的入库记录。
- 从具体入库批次发起采购退货，并跳转到进货/退货页的退货 Tab。
- 支持历史库存或期初库存补录，通过 `入库来源` 区分普通采购入库和历史库存入库。

### 5. 货物状态与资金状态口径

建议把进货单列表的 `状态` 列改为 `货物状态`，把 `资金汇总` 列改为 `资金状态`。

货物状态只关心采购货物有没有下单、有没有入库：

| 货物状态 | 含义 | 进入条件 |
| --- | --- | --- |
| 预下单 | 采购单已创建，但还没有真正向商家确认下单 | 新增进货单默认状态 |
| 已下单 | 已向商家下单，等待到货 | 预下单确认后进入 |
| 部分入库 | 采购商品只入库了一部分 | 已下单后发生部分到货入库 |
| 已入库 | 商品已经全部入库 | 入库数量满足采购数量 |
| 已取消 | 预下单阶段取消 | 取消操作二次确认后进入 |

资金状态建议单独展示和筛选：

```text
未付款 / 部分付款 / 已结清
```

这样可以避免把“货是否到齐”和“钱是否结清”混成一个模糊状态。

`已完成` 不再作为独立状态使用。历史上的 `已完成` 可迁移或展示为 `已入库`：

```text
原已完成 -> 已入库
```

采购退货不改变进货单的货物状态。对进货单来说，退货流程属于入库记录和库存件维度；进货单仍然只关心货是否已经按原采购流程入库，以及资金是否结清。

### 6. 资金计算口径

进货单和采购退货记录的资金边界需要拆清楚：

- 进货单只记录采购本身的付款，不记录退货、退款或退货收款。
- 采购退货记录只记录退货本身的资金状态，不回写进货单资金记录。
- 进货单资金状态只回答“这张进货单自身有没有付款结清”。
- 退货资金状态只回答“这张退货单自身有没有收到退款”。

进货单资金状态建议从金额推导，不手动维护：

```text
应付金额 = 已入库商品金额 + 实际运费
已付金额 = 有效付款记录合计
未结清金额 = 应付金额 - 已付金额
```

资金状态建议：

```text
未付款   已付金额 = 0 且未结清金额 > 0
部分付款 已付金额 > 0 且未结清金额 > 0
已结清   未结清金额 = 0
```

采购退货本身不改变进货单货物状态，也不改变进货单资金状态；退货金额和退款/收款状态在采购退货记录里独立维护。

### 7. 进货/退货页与入库单页面协作方式

进货/退货页作为采购工作台，不直接承接入库执行；入库单页面作为入库执行页，承接新增入库和入库记录查询。

采购退货的入口可以出现在入库单页面，但新增退货 Modal 和退货记录维护统一放在进货/退货页的退货 Tab 中处理。

物流公司、物流单号和运费记录作为底层能力已在 `plans/done/logistics-management-plan.md` 归档，进货和退货流程后续只负责选择并填写物流信息。

建议跳转规则：

```text
点击“入库”
-> /admin/dashboard/warehouse/inbound?purchaseOrderId={purchaseOrderId}&action=create
-> 入库单页面拉取进货单数据，自动打开新增入库单 Modal，并回填商家、商品明细、剩余可入库数量、运费等信息

点击“查看入库记录”
-> /admin/dashboard/warehouse/inbound?purchaseOrderId={purchaseOrderId}
-> 入库单页面按进货单筛选入库记录

点击“退货”
-> /admin/dashboard/warehouse/purchase-orders?tab=returns&inboundOrderId={inboundOrderId}&action=create
-> 进货/退货页切换到退货 Tab，按入库单拉取可退数据，自动打开新增退货 Modal 并回填

点击“查看退货记录”
-> /admin/dashboard/warehouse/purchase-orders?tab=returns&inboundOrderId={inboundOrderId}
-> 进货/退货页切换到退货 Tab，并按入库单筛选采购退货记录
```

参数建议优先使用 `purchaseOrderId` 和 `inboundOrderId`，订单号只用于页面展示，不作为主要查询条件。

## 进货单页建议交互

### 列表层

进货单列表继续展示：

- 进货单号
- 商家
- 货物状态
- 商品汇总
- 资金状态
- 下单时间
- 操作

操作建议：

- `预下单`：显示“确认下单”“取消”，两个操作都需要二次确认
- `已下单`：显示“编辑”“入库”；点击“入库”跳转到入库单页面并自动打开新增入库 Modal
- `部分入库`：显示“编辑”“继续入库”“查看入库记录”；“继续入库”和“查看入库记录”都跳转到入库单页面
- `已入库`：显示“编辑”“查看入库记录”；资金状态不是 `已结清` 时展示“付款”
- `已取消`：显示“查看”

其中“付款”也可以放在资金状态区域或详情层，列表操作列优先保持简洁。

### 编辑边界

进货单编辑需要根据货物状态限制可编辑范围，避免已经入库后的库存成本被反向改乱。

建议：

- `预下单`、`已下单`：允许编辑商家、商品、采购数量、采购价、预估运费、备注。
- `部分入库`：不允许修改已入库明细；第一阶段建议只允许编辑备注、预估运费等辅助信息，采购差异通过后续入库或退货处理。
- `已入库`：只允许编辑备注、实际运费等辅助信息，不允许修改商品、数量和采购价。
- `已取消`：只读查看。

### 表单和入库弹窗

新增进货单或编辑未完成进货单时，运费字段 label 建议展示为 `预估运费`。

进货单进入已入库后，运费字段 label 建议展示为 `实际运费`。

运费数据仍然挂在进货单上，不挂在入库单上。入库单页面展示和修改运费只是为了方便用户在到货时顺手修正，保存时仍回写到进货单。

入库单页面被 `action=create` 参数打开时，新增入库 Modal 建议：

- 回显进货单上的运费；未全部入库前每次入库都回显该值，允许用户在入库时修正。
- 字段 label 可展示为 `实际运费`，但数据仍保存到进货单。
- 底部增加付款信息，字段和付款 Modal 保持一致：付款金额、付款账号、付款时间。
- 付款金额默认 `0`，为 `0` 时只入库不创建付款记录；大于 `0` 时入库成功后同步创建付款记录。
- 如果付款后资金状态达到已结清，且商品已经全部入库，则货物状态为 `已入库`，资金状态为 `已结清`。

### 详情层

进货单详情当前通过列表操作里的详情 icon 展开，点击后增加行高展示详情。这个交互先不调整，不影响主流程。

详情分块展示：

- 商品明细
- 入库记录
- 付款记录
- 备注

其中“入库记录”每条展示：

- `RK-{id}`
- 入库时间
- 入库数量
- 入库成本
- 入库记录状态
- 操作：查看，点击后跳转到入库单页面查看该进货单的入库记录

采购退货不建议放在进货单详情里直接提交，统一放到入库单页面的具体入库记录上处理。

## 入库单页建议交互

入库单页面建议从独立“新增入库单页面”逐步调整为“入库记录页面 + 入库执行页面”：

- 从进货单点击“入库”进入时，自动打开新增入库 Modal。
- 从进货单点击“查看入库记录”进入时，自动按该进货单筛选记录。
- 退货发起操作可以在入库单页面展示，并且必须落到具体入库记录。
- 点击“退货”和“查看退货记录”都走跳转逻辑，目标是进货/退货页的退货 Tab。
- 货物状态、资金状态、退货运费和退货物流的后续维护放在采购退货记录里。

列表建议展示：

- 入库单号
- 关联进货单号
- 商家
- 入库来源
- 入库时间
- 商品汇总
- 入库数量
- 已售数量
- 在库数量
- 已退数量
- 可退数量
- 入库记录状态
- 操作

操作建议：

- `已入库`：显示“详情”“退货”
- `部分退货`：显示“详情”“退货”“查看退货记录”
- `已退货`：显示“详情”“查看退货记录”，不展示“退货”

### 新增退货 Modal

当前“一起退货”的能力可以保留为快捷入口，但默认交互建议改为“选择退货”。

原因：

- 一张入库单可能包含多种商品。
- 同一种商品可能有多件库存件，且部分已经售出。
- 采购退货只能退仍在库的库存件，不能退已售出的库存件。
- 整批退货是低频场景，默认整批会增加误操作风险。

新增退货可以从两个入口进入：

```text
入库单操作列点击“退货”
-> 带 inboundOrderId 跳转到退货 Tab
-> 自动打开新增退货 Modal，并回填进货单、入库单、商家和可退商品

退货 Tab 点击“新增退货”
-> 打开新增退货 Modal
-> 用户先选择进货单号或入库单号，再回填关联数据
```

新增退货 Modal 的单据定位建议：

- 进货单号和入库单号二选一作为起点。
- 选择进货单号后，自动回填商家，并加载该进货单下可退的入库单列表；用户需要再选择具体入库单。
- 选择入库单号后，自动反查并回填进货单号和商家；进货单号禁用。
- 最终必须确定具体入库单后，才能加载可退商品和可退库存件。

退货商品选择建议：

- 按商品分组展示入库数量、已售数量、已退数量、可退数量。
- 如果该商品开启了序列号管理，退货时必须选择具体库存件或序列号。
- 如果该商品未开启序列号管理，可以按商品填写退货数量，由系统自动选择可退库存件。
- 默认只允许选择 `in_stock` 的库存件。
- 支持整批退货快捷操作，但执行前需要二次确认。
- 提交后创建采购退货记录，并把选中的库存件状态改为 `returned`。
- 新增退货 Modal 只负责发起退货，不负责维护后续货物状态和资金状态。
- 退款允许延迟发生，后续在进货/退货页的退货 Tab 中维护。
- 退货金额不回写进货单资金状态，采购退货记录维护自己的资金状态。

## 采购退货 Tab 建议交互

`进货/退货` 页中的 `退货` Tab 用于维护已经创建的采购退货记录。

退货记录从入库单页面发起创建，创建后进入退货 Tab 继续维护。

退货 Tab 顶部建议提供“新增退货”按钮。点击后打开新增退货 Modal，用户先选择进货单号或入库单号，再回填商家、关联单据和可退商品。

列表建议展示：

- 退货单号
- 关联进货单号
- 关联入库单号
- 商家
- 商品汇总
- 退货金额
- 退货运费
- 运费承担方
- 货物状态
- 资金状态
- 创建时间
- 操作

退货 Tab 中：

```text
货物状态 = 退货货物状态，用于表达退货商品是否已寄回、商家是否已收货
资金状态 = 退款状态，用于表达退款是否收到
```

操作列建议根据两个状态共同决定：

- 货物状态 `待寄回`：显示“编辑”“确认发货”“取消”
- 货物状态 `已寄回`：显示“编辑”“确认商家收货”
- 货物状态 `商家已收货`：显示“编辑”
- 货物状态 `已取消`：显示“查看”
- 货物状态不是 `已取消` 且资金状态不是 `已退款`：额外显示“确认收款”
- 资金状态是 `已退款`：不展示“确认收款”

也就是说，货物状态决定退货物流相关操作，资金状态决定收款相关操作；“确认收款”表示确认已经收到商家的退款。

资金状态不直接决定退货货物状态。退款可以延迟发生：

```text
货物状态 = 商家已收货
资金状态 = 未退款 / 部分退款
```

这种状态组合是允许存在的。

退货 Tab 搜索建议：

- 退货单号
- 进货单号
- 入库单号
- 商家
- 货物状态：全部、待寄回、已寄回、商家已收货、已取消
- 资金状态：全部、未退款、部分退款、已退款
- 创建时间范围

商品选择不作为新增退货前置搜索；新增退货时，先确定进货单或入库单，再从对应入库单的可退商品明细里选择退货商品。

换货场景已拆到 `plans/backlog/exchange-management-backlog.md`，当前采购主流程暂不实现独立换货单。

第一阶段用：

```text
退货原商品 -> 新商品重新入库
```

后续如果换货高频，再从 backlog 候选计划中评估独立换货单，承接原商品寄回、新商品补发、差价、运费和序列号替换。

## 入库记录状态

入库记录状态不建议人工维护，建议从库存件状态推导：

```text
已入库   returned_quantity = 0
部分退货 0 < returned_quantity < inbound_quantity
已退货   returned_quantity = inbound_quantity
```

需要注意：

- 已售出的库存件不能采购退货。
- 如果一张入库单部分商品已售、剩余商品已退，则状态仍应是“部分退货”，并额外显示可退数量。
- 页面上可以同时展示 `入库数量 / 已售数量 / 在库数量 / 已退数量`，列表里也可以简化展示为 `已售 / 在库 / 已退`，避免状态含义不清。
- 操作列只有 `已入库` 和 `部分退货` 展示“退货”，`已退货` 不再展示“退货”。
- 只有 `部分退货` 和 `已退货` 展示“查看退货记录”。

## 搜索与筛选

如果保留独立入库记录页，建议增加搜索表单：

- 进货单号
- 商家
- 商品关键词
- 入库记录状态：全部、已入库、部分退货、已退货
- 入库时间范围

从进货单页带 `purchaseOrderId` 跳转过来时，搜索表单自动回填并锁定或高亮该进货单筛选条件；用户仍可以手动清空筛选，回到全局入库记录列表。

如果入库记录主要放在进货单详情里，则独立入库记录页的搜索可以后置；进货单页本身已经有商家/状态/关键词筛选的基础。

进货单页搜索建议：

- 货物状态默认查询 `预下单`、`已下单`、`部分入库`。
- 资金状态默认查询 `未付款`、`部分付款`。
- `已入库` 且 `已结清` 的进货单默认不出现在待办视图中，需要用户主动切换筛选查看。

## 落地优先级建议

### 第一阶段：不大动数据结构，先顺流程

- 在进货单详情中展示关联入库记录。
- 进货单的“入库”“查看入库记录”通过 URL 参数跳转到入库单页面。
- 采购退货从入库单页面的具体入库记录发起，并跳转到进货/退货页的退货 Tab 处理。
- 采购退货记录放在进货/退货页的退货 Tab 中维护。
- 新增退货 Modal 从“一起退货”升级为“选择退货”，保留整批退货快捷操作。
- 进货单状态列改名为 `货物状态`，资金汇总列改名为 `资金状态`。
- 进货单增加 `预下单`、`已入库` 货物状态；不再保留 `已完成` 状态。
- 入库单页面的新增入库弹窗支持修正实际运费，并可选同步记录付款。
- 入库记录增加推导状态：已入库、部分退货、已退货。
- 入库单页面增加搜索表单，支持按进货单号、商家、商品关键词、入库记录状态、入库时间筛选。
- 退货 Tab 增加搜索表单，支持按退货单号、进货单号、入库单号、商家、货物状态、资金状态、创建时间筛选。
- 入库单菜单保留原名，页面承接入库记录查询、采购入库和历史库存入库。

### 第二阶段：优化查询和展示

- 入库记录接口支持商家和入库记录状态筛选。
- 入库记录页增加入库数量、已售数量、在库数量、已退数量、可退数量汇总。
- 独立入库记录页只作为全局流水查询。

### 第三阶段：优化历史库存

等实际使用路径稳定后，再决定：

- 历史库存入库是否需要更明显的快捷入口。
- 历史库存入库是否需要独立模板或批量导入能力。
