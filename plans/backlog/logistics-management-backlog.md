# 物流信息与运费管理计划

## 背景

进货单里需要记录运费，但运费本身又有独立统计价值。例如同一段时间在京东、顺丰、德邦等物流渠道上分别花了多少钱。

因此运费不宜只作为进货单上的一个孤立金额字段，需要先拆出一个轻量的物流信息模块，用于维护物流公司和运费记录。

这个模块作为底层能力，后续进货和退货都可以直接选择物流公司、填写物流单号和运费。

当前第一至第四阶段已落地，后续重点进入批量结算和更细的财务口径打磨。

- 当前不做历史单据回填；现阶段数据均为测试数据，新流程从当前版本开始生效。

- 物流付款当前按单条记录结清；按物流公司批量结清暂未实现。

## 当前卡点与待确认

当前没有技术阻塞，主要是后续范围需要排期：

- 进货单阶段不强制物流公司。
  供应商发货时可能只提供物流单号，不一定同步物流公司，因此进货单新增/编辑允许只填物流单号。
- 入库确认阶段强制物流公司。
  到货入库时我方已经知道实际物流公司和花销，因此进货单入库时必须选择物流公司。
- 不做历史数据回填。
  当前都是测试数据，历史进货和历史退货不作为正式统计基础。
- 多包裹场景不拆多条记录。
  现阶段只看物流单号，不做一张业务单据多条物流记录。
- 物流成本单列统计。
  物流成本已经在物流管理页单列汇总，不直接混入销售利润主口径；后续可以在利润分析里作为独立成本维度引用。
- 批量付款是否需要做？
  当前可以单条物流记录付款。如果实际结算是按物流公司月结，建议后续做按公司批量结清。

## 下一步建议

建议下一步按这个顺序做：

1. 根据实际月结方式决定是否做按物流公司批量付款。
2. 设计已付款物流记录被业务取消后的反冲或退款处理。
3. 观察真实业务里的多包裹比例；如果只看物流单号，当前一单一记录模型可以继续保留。
4. 再考虑利润/成本看板是否引用这组物流成本数据。
5. 如果统计页使用频率高，再增加从统计结果反查物流明细的入口。

## 当前判断

建议把物流信息模块拆成两层：

- 物流公司：维护京东、顺丰、德邦等物流渠道基础信息。
- 物流记录：记录某次物流产生的单号、运费、业务关联和结算口径。

进货单上的运费仍然保留，但需要拆清两个口径：

```text
采购成本 = 货款 + 我方实际承担的采购相关运费
应付对象 = 商家应付、物流公司应付分别计算
```

例如货款 2000，运费 20：

- 20 元由我方直接付物流公司。
- 采购成本是 2020。
- 商家待付款是 2000。
- 物流待付款是 20。

因此物流费用不能天然并入“商家待付款”，需要单独进入物流待付款。

当前阶段暂不考虑商家代收运费场景。进货物流默认由我方承担并支付给物流公司；采购退货物流只处理我方承担、商家承担、双方平摊三种场景。

入库单页面展示运费只是为了方便用户在到货时修正；数据仍然回写到进货单，不挂在入库单上。

- 备注

### 2. 物流记录 CRUD

建议增加物流记录管理能力：

- 新增物流记录
- 编辑物流记录
- 删除或作废物流记录
- 查询物流记录

字段建议：

- 物流类型：进货物流、采购退货物流、手工记录
- 物流公司
- 物流单号，可选
- 运费金额
- 我方承担金额
- 发生时间
- 关联单据，可选：进货单、入库单、采购退货单
- 运费承担方：我方、商家、平摊

- 付款状态：未付款、已付款、作废
- 付款时间，可选
- 付款账户，可选
- 备注

### 3. 与进货单运费的关系

进货单上的运费是采购流程里的成本字段，不应默认等同于商家应付字段。

物流模块里的物流记录同时承担三个作用：

- 记录物流单号和物流公司，方便业务追踪。
- 记录运费发生额和我方承担额，方便统计采购成本。
- 记录结算对象和付款状态，方便财务拆分商家待付款和物流待付款。

```text
进货单运费 = 用于采购成本
物流记录 = 用于物流追踪、费用统计、物流应付管理
商家应付 = 货款，不包含我方直接付给物流公司的运费
物流应付 = 我方承担且未付款的物流费用
```

第一阶段建议直接完成物流公司、物流记录和物流待付款能力，再接入进货和退货流程。

后续可以在入库、编辑进货单或新增退货时增加“关联物流信息”：

- 选择物流公司。
- 填写物流单号。
- 填写运费。
- 选择运费承担方和结算对象。
- 保存后回写对应业务单据的成本运费，并创建或关联一条物流记录。

### 4. 搜索与统计

物流费用列表建议支持：

- 物流类型：进货、退货、手工记录
- 物流公司
- 关联单据
- 物流单号
- 结算对象
- 付款状态
- 时间范围

统计建议：

- 按物流公司汇总运费。
- 按月份汇总运费。
- 按物流类型汇总进货运费和退货运费。
- 按结算对象汇总待付款金额。
- 查看某个进货单或采购退货单关联的物流费用。
- 财务管理中展示物流待付款，避免进货商家应付被运费放大。

### 5. 财务汇总适配

账款管理的汇总口径需要同步调整，不能只改明细列表。

当前账款汇总需要拆成四组：

- 商家待付款：只统计进货货款和明确由商家收取的费用，不包含我方直接付给物流公司的运费。
- 物流待付款：统计物流记录中我方承担且未付款的金额。
- 供应商待退款：统计采购退货后商家应退金额。
- 客户待收款：统计销售订单未收金额。

顶部汇总卡片建议从当前的三项调整为：

```text
商家待付款
物流待付款
待退款
客户待收款
```

汇总视图建议增加物流维度：

- 商家汇总：按进货商家汇总商家待付款。
- 物流汇总：按物流公司汇总物流待付款。
- 客户汇总：按客户汇总销售待收款。

分笔明细建议增加物流待付款 Tab：

- 进货应付/退款：只展示商家应付和供应商退款。
- 物流待付款：展示未付款物流记录，可登记物流付款。
- 销售待收：展示客户未收款订单。

接口汇总字段也需要拆分：

```text
summary.merchant_payable_count
summary.merchant_payable_amount
summary.logistics_payable_count
summary.logistics_payable_amount
summary.refund_count
summary.refund_amount
summary.receivable_count
summary.receivable_amount
```

其中 `merchant_payable_amount + logistics_payable_amount` 才是总待付款，但页面上应保持分开展示。

### 第二阶段：接入进货流程（已完成）

- 进货单继续保留运费字段。
- 入库 Modal 回显并允许修正进货单运费。
- 入库或编辑进货单时可以选择物流公司、填写物流单号和运费。
- 进货运费参与采购成本统计。

### 第三阶段：接入退货流程（已完成）

- 新增退货时可以选择物流公司、填写物流单号和退货运费。

### 第四阶段：运费统计（已完成）

- 支持关联进货单或退货单。
- 支持按物流公司、时间范围统计运费。

- 暂未做从统计结果反查到具体进货单、入库单或采购退货单。

- 物流付款登记当前按单条物流记录结清；按物流公司批量结清暂未实现。
- 财务首页或销售数据页是否引用物流成本拆分，需要后续单独设计。
