# 路由切换体验优化方案

## 1. 问题诊断

在 Next.js App Router 中，当点击一个链接跳转到新路由时，如果目标页面是服务端渲染（SSR）或者包含较重的服务端组件（Server
Components），Next.js 会在后台完成数据获取和组件渲染后才进行 DOM 替换。
**现象**：点击链接后，页面看起来没有任何反应（停顿），直到新页面准备就绪才突然跳转。
**原因**：缺少客户端的即时反馈机制（如进度条）以及服务端渲染过程中的占位符（loading.tsx）。

---

## 2. 优化方案

### 方案一：引入全局顶部进度条 (推荐)

这是解决“点击无反应”最直接有效的方法。通过在全局布局中添加一个顶部进度条，用户在点击链接的瞬间就能看到进度反馈。

- **工具**：`nextjs-toploader`
- **实施**：
    1. 安装依赖：`npm install nextjs-toploader`
    2. 在 `app/layout.tsx` 中引入并配置。
- **优点**：零配置实现全局反馈，符合用户对 Web 应用的直觉。

### 方案二：利用 Next.js `loading.tsx` 机制

在 App Router 中，`loading.tsx` 可以自动为路由段创建 Suspense 边界。

- **实施**：
    1. 在 `app/` 根目录下创建 `loading.tsx`，显示一个全局的加载动画或 Antd 的 `Spin`。
    2. 在 `app/admin/dashboard/` 等核心业务目录下创建特定的 `loading.tsx`，显示骨架屏（Skeleton）。
- **优点**：能够让用户快速进入新页面结构，减少“白屏”或“卡顿”感。

### 方案三：优化数据获取策略

- **预取 (Prefetching)**：Next.js 默认会预取视口内的 `<Link>`。确保没有禁用此功能。
- **流式渲染 (Streaming)**：将页面中耗时的数据获取逻辑拆分到独立的客户端组件或使用 `Suspense` 包裹，让页面主体先渲染。

---

## 3. 实施计划

1. **安装依赖**：安装 `nextjs-toploader`。
2. **配置全局进度条**：修改 `app/layout.tsx`。
3. **创建基础加载页**：编写 `app/loading.tsx`。
4. **创建业务骨架屏**：为 `admin/dashboard` 编写更精细的 `loading.tsx`。

---

## 4. 预期效果

- 用户点击链接后，顶部立即出现进度条。
- 页面结构快速切换到加载状态（骨架屏），而不是停留在旧页面。
- 整体感知速度提升，消除“系统卡死”的错觉。
