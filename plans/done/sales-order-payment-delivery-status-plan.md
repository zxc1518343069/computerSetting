## 背景

当前后台销售订单列表包含三个状态相关列：

- `库存状态`：根据订单明细商品的当前在库数量推导，展示库存是否满足。
- `收款`：当前由 `sales_orders.is_paid` 表达，只有已收款和未收款。
- `状态`：当前由 `sales_orders.status` 表达，取值为 `pending`、`completed`、`cancelled`。

现有 `status` 的真实含义更接近“结算/扣库状态”：

- `pending`：待结算，可编辑、可调整配置、可结算。
- `completed`：已结算，结算接口已绑定具体库存件、扣减库存、写入 `sold_at`。
- `cancelled`：已取消。

但从业务语言看，“状态”容易被理解成订单整体生命周期，和收款、库存、交付混在一起。后续建议拆成更清楚的两个人工业务状态：收款状态和交付状态。

## 结论草案

销售订单不再强调一个泛化的“订单状态”，而是使用两个明确维度：

```text
收款状态 payment_status
交付状态 delivery_status
```

库存状态继续作为系统推导值，不作为人工流转状态：

```text
库存状态 = 当前库存是否满足未交付订单的出库需求
```

## 收款状态

建议第一版使用四个值：

| 状态值 | 页面文案 | 含义 |
| --- | --- | --- |
| `unpaid` | 未收款 | 未收到客户款项 |
| `paid` | 已收款 | 已收到客户款项 |
| `refund_pending` | 待退款 | 订单已取消，但已收款项尚未退还 |
| `refunded` | 已退款 | 订单取消后，已收款项已退还 |

不建议继续只使用 `is_paid`，因为它无法表达“待退款”和“已退款”。

## 交付状态

建议第一版使用三个值：

| 状态值 | 页面文案 | 含义 |
| --- | --- | --- |
| `undelivered` | 未交付 | 尚未把整机或配件交给客户 |
| `delivered` | 已交付 | 已绑定库存件、扣减库存，并交付给客户 |
| `cancelled` | 已取消 | 订单取消，不再交付 |

当前“结算”动作更适合改名为“确认交付”或“交付”，因为它实际完成的是：

```text
选择具体库存件 -> 扣减库存 -> 记录成本和利润 -> 订单进入已交付
```

## 合法状态组合

建议只允许以下组合：

| 收款状态 | 交付状态 | 含义 |
| --- | --- | --- |
| 未收款 | 未交付 | 正常待处理订单 |
| 已收款 | 未交付 | 已收款，等待装机或交付 |
| 未收款 | 已交付 | 已交付但客户尚未付款，属于应收款 |
| 已收款 | 已交付 | 正常完成订单 |
| 未收款 | 已取消 | 未收款订单取消 |
| 待退款 | 已取消 | 已收款订单取消，但款项尚未退还 |
| 已退款 | 已取消 | 已收款订单取消，且款项已退还 |

原则上不建议出现：

- `已退款 + 未交付`
- `已退款 + 已交付`
- `待退款 + 未交付`
- `待退款 + 已交付`
- `已收款 + 已取消`

如果已收款订单被取消，应在取消时直接落到 `待退款 + 已取消` 或 `已退款 + 已取消`，避免“已收款 + 已取消”这种含义不完整的中间状态。

## 操作列建议

订单列表操作列建议调整为：

```text
编辑 | 调整配置 | 确认交付 | 取消
```

按钮建议统一使用“图标 + 文案”，不要只放纯图标按钮。状态动作涉及资金、库存和交付，只靠图标理解成本较高。

操作边界：

| 操作 | 允许条件 | 影响 |
| --- | --- | --- |
| 编辑 | 未交付且未取消 | 客户信息、成交金额、收款状态、备注 |
| 调整配置 | 未交付且未取消 | 调整实际装机配置和最终成交金额 |
| 确认交付 | 未交付且未取消，且库存满足 | 绑定库存件、扣库存、计算成本利润，交付状态变为已交付 |
| 取消 | 未交付且未取消 | 交付状态变为已取消，并根据收款状态处理退款状态 |

操作列不能所有按钮一直展示，应按状态动态展示：

| 当前状态 | 操作列展示 |
| --- | --- |
| 未收款 + 未交付 | 编辑、调整配置、确认交付、取消 |
| 已收款 + 未交付 | 编辑、调整配置、确认交付、取消 |
| 未收款 + 已交付 | 不展示编辑、调整配置、取消；后续如有需要进入销售退货流程 |
| 已收款 + 已交付 | 不展示编辑、调整配置、取消；后续如有需要进入销售退货流程 |
| 未收款 + 已取消 | 不展示业务操作 |
| 待退款 + 已取消 | 展示“标记已退款” |
| 已退款 + 已取消 | 不展示业务操作 |

库存不足时，“确认交付”不应允许点击。实现上可以保留按钮但禁用并用 Tooltip 说明“库存不足，无法交付”，也可以隐藏该操作；推荐保留禁用态，用户更容易理解为什么暂时不能交付。

第一版不建议允许已交付订单直接“取消”。已交付后的撤销应该进入销售退货、换货或售后流程，涉及库存回滚、退款记录和利润重算，不能和未交付取消混为一个按钮。

## 列表筛选与默认视图

订单列表需要增加交付状态筛选项。建议筛选区包含：

| 筛选项 | 选项 |
| --- | --- |
| 关键词 | 订单号、客户名称、手机号 |
| 收款状态 | 全部、未收款、已收款、待退款、已退款 |
| 交付状态 | 全部、未交付、已交付、已取消 |
| 订单范围 | 待办订单、全部订单 |

默认进入订单列表时，建议使用“待办订单”范围，避免已完成订单干扰日常处理。

“待办订单”建议包含仍需处理或跟进的订单：

| 状态组合 | 是否进入默认待办 |
| --- | --- |
| 未收款 + 未交付 | 是 |
| 已收款 + 未交付 | 是 |
| 未收款 + 已交付 | 是，属于应收款 |
| 待退款 + 已取消 | 是，属于待退款 |
| 已收款 + 已交付 | 否，正常完成 |
| 未收款 + 已取消 | 否，已取消且无资金后续 |
| 已退款 + 已取消 | 否，已取消且退款完成 |

这样默认列表会过滤掉 `已收款 + 已交付` 这类已经闭环的订单，但用户仍可通过“全部订单”或明确筛选 `收款状态 = 已收款`、`交付状态 = 已交付` 查回历史订单。

如果第一版不想引入“订单范围”筛选，也可以用一个开关：

```text
隐藏已闭环订单
```

推荐仍使用“订单范围：待办订单 / 全部订单”，因为文案更符合业务语义，也方便后续扩展“异常订单”“售后订单”等视图。

## 订单明细展开区

当前展开区同时展示：

```text
原始下单配置
当前实际装机配置
```

建议调整展示规则，减少未处理订单的重复信息。

如果订单未交付且没有配置调整：

```text
只展示原始下单配置
```

此时“当前实际装机配置”与原始配置完全一致，额外展示一块会增加阅读成本。

如果订单未交付但已经调整过配置：

```text
展示原始下单配置 + 当前实际装机配置 + 最近调整信息
```

因为这时两者已经产生业务差异，需要同时保留报价快照和实际准备装机配置。

如果订单已交付：

```text
展示原始下单配置 + 实际交付配置
```

已交付订单的第二块文案建议从“当前实际装机配置”改为“实际交付配置”，更准确表达它已经绑定库存并出库。

## 后端状态校验

状态机必须以后端校验为准，前端显隐只作为交互优化。

接口层需要禁止非法状态组合，包括但不限于：

- `已退款 + 未交付`
- `已退款 + 已交付`
- `待退款 + 未交付`
- `待退款 + 已交付`
- `已收款 + 已取消`

后端取消、确认交付、标记收款、标记退款等接口，都应该统一走状态流转函数，避免多个接口各自拼状态导致组合失控。

## 关键操作审计字段

取消订单和退款标记都需要留下可追溯信息。

建议销售订单补充：

```text
cancelled_at
cancelled_by_user_id
cancelled_by_username
cancel_reason
refunded_at
refunded_by_user_id
refunded_by_username
refund_note
```

第一版可以先不做完整退款流水，但至少要记录“谁在什么时候标记退款完成”和备注。后续如果增加退款流水，再将 `refunded_at` 等字段作为冗余快照或由流水汇总。

## 确认交付与未收款

建议允许未收款订单确认交付，用于支持赊账或后收款场景。

确认交付弹窗需要展示当前收款状态。如果订单仍是未收款，应给出明确提示：

```text
该订单尚未收款，交付后将进入应收款。
```

不建议在确认交付时强制改为已收款。收款是资金动作，交付是履约动作，两者应保持独立。

## 库存状态展示规则

库存状态只对未交付订单有业务意义。

建议展示规则：

| 交付状态 | 库存状态列展示 |
| --- | --- |
| 未交付 | 库存满足 / 库存不足 |
| 已交付 | 已出库 |
| 已取消 | - |

原因是已交付订单已经扣减库存，再展示当前库存是否满足会误导用户；已取消订单不再需要库存履约。

## 财务与账款联动

订单页状态调整后，财务和账款页也需要同步语义。

建议账款页按组合识别待办：

| 状态组合 | 财务含义 |
| --- | --- |
| 未收款 + 未交付 | 待收款，但尚未交付 |
| 未收款 + 已交付 | 应收款，优先级更高 |
| 待退款 + 已取消 | 应退款待办 |
| 已收款 + 已交付 | 已闭环 |
| 未收款 + 已取消 | 已闭环 |
| 已退款 + 已取消 | 已闭环 |

如果第一版暂时不改账款页，至少订单 API 返回的新状态要能支持后续聚合，避免继续依赖 `is_paid` 和 `status != cancelled` 这种粗粒度判断。

## 兼容期策略

当前已有字段：

```text
status
is_paid
sold_at
```

建议第一版新增字段并进入兼容期，不立刻删除旧字段：

```text
payment_status
delivery_status
delivered_at
```

兼容期内：

- 新页面优先使用 `payment_status` 和 `delivery_status`。
- 旧字段可由新字段同步写入，维持历史接口和导出兼容。
- 后续确认没有旧逻辑依赖后，再考虑清理或停止使用 `status/is_paid`。

默认筛选也要在 UI 上明确显示当前范围，例如：

```text
订单范围：待办订单
```

避免用户以为历史已完成订单消失。

## 取消订单流程

订单列表应新增“取消”按钮。

### 未收款订单

如果当前收款状态为 `unpaid`：

```text
取消 -> 未收款 + 已取消
```

弹窗只需要确认是否取消订单，并可选填写取消原因。

### 已收款订单

如果当前收款状态为 `paid`：

取消弹窗需要提醒：

```text
该订单已收款。取消前请确认是否已经将款项退还给客户。
```

弹窗提供一个勾选项：

```text
已将款项退还给客户
```

提交规则：

| 勾选项 | 取消后的状态 |
| --- | --- |
| 未勾选 | 待退款 + 已取消 |
| 已勾选 | 已退款 + 已取消 |

这样可以避免取消已收款订单时遗漏退款责任。

### 待退款订单

如果订单已经是 `待退款 + 已取消`，后续应提供独立的“标记已退款”入口：

```text
待退款 + 已取消 -> 已退款 + 已取消
```

这个入口可以放在订单列表、财务应收页，或后续退款记录模块中。第一版如果不做退款流水，可以先做人工标记。

## 与当前字段的迁移关系

当前字段：

```text
status: pending | completed | cancelled
is_paid: boolean
sold_at: datetime | null
```

建议目标字段：

```text
payment_status: unpaid | paid | refund_pending | refunded
delivery_status: undelivered | delivered | cancelled
delivered_at: datetime | null
cancelled_at: datetime | null
cancel_reason: string | null
```

历史数据可按以下规则迁移：

| 当前字段 | 新状态 |
| --- | --- |
| `status = pending` 且 `is_paid = 0` | 未收款 + 未交付 |
| `status = pending` 且 `is_paid = 1` | 已收款 + 未交付 |
| `status = completed` 且 `is_paid = 0` | 未收款 + 已交付 |
| `status = completed` 且 `is_paid = 1` | 已收款 + 已交付 |
| `status = cancelled` 且 `is_paid = 0` | 未收款 + 已取消 |
| `status = cancelled` 且 `is_paid = 1` | 待退款 + 已取消 |

`sold_at` 可继续作为历史兼容字段，但页面语义建议改为 `delivered_at`。
